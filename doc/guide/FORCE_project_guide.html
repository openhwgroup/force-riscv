FORCE-RISCV project guide

1  Coding standard

Applying a coding standard is very helpful when there are multiple contributors to the project, so that all contributed code has similar look and feel.

This is a standard found on the web that seems to make quite a lot of sense: http://www.possibility.com/Cpp/CppCodingStandard.html.

Like the author said: It helps if the standard annoys everyone in some way so everyone feels they are on the same playing field.

The coding standard is also adopted by CMU ECE department: https://users.ece.cmu.edu/~eno/coding/CppCodingStandard.html

We are adopting from the original standard, while narrowed down on things like file name extensions (*.h and *.cc).

In order to view this standard from within the repository, an amended version of the above page is saved/checked-in in the guide directory, you can view it here:

CppCodingStandard.html.

Additionally, both header and source file of a module should be wrapped in Force namespace.

Please refer to base/inc/Config.h and base/sr/Config.cc as examples of applying the coding style.

3  Utilities

Utilities are built over time to facilitate code development in the project.

3.1 Logging

Always provide sensible logging information in the code to help debugging/tracing.  Also try and strike a balance, dont provide too much logging information.  We can start with fewer logging lines and add more as needed.

Currently there are 7 logging levels, they are listed here from most verbose to less verbose: trace, debug, info, warn, error, fail, notice.

3.1.1 Logging level switch

By default the logging level is set at error, which means only logging statements with error, fail, notice will be shown.  The user can specify a different logging level on the command line using -l switch.

For example: bin/friscv -l debug will setup logging level to debug and the log will be more verbose than default.

3.1.2 Logging syntax

A logging statement is in this format LOG(level) << out-put-stream.

For example:

LOG(trace) << The user has specified config file \ << file_name << \. << endl;

3.1.3 Notice logging level

Notice level will always show up in the log, so dont use it extensively.

Example:

LOG(notice) << Initial seed = 0x << hex << seed << endl;

3.1.4 Error logging level

Error logging is for reporting error but not immediately exit the program.  There are two types of handling after an error, either a variable containing the error state is passed back to the caller to handle, or an exception is thrown to be caught by proper exception handling code in the hierarchy.  Also we usually include the function name where the error occur in the error message.

For example:

LOG(error) << {parse_bin32} Error parsing \ << str << \, invalid argument. << endl;

*error_status = true;

3.1.5 Fail logging level

Fail logging level is used when there is non-recoverable error and we want the program to exit immediately.  So there will be a LOG(fail) statement followed by a FAIL statement.  It is not necessary to include function name etc in the error message, because the FAIL statement will include function name, file name, line number etc.  If the logging level is set to more verbose than error, we assume the user wants to debug, and the FAIL statement will try to dump core.

In the FAIL statement, try to give a short, unique string as a brief description of the fail type.  This will help in the binning of fail signatures.

Example of fail logging usage in code:

LOG(fail) << Unknown logging level \ << logLevel << \. << endl;

FAIL(unknown-log-level);

To observe a fail report in action, try this:

cd force-riscv

bin/friscv -l abcd -s 1234

This will fail out due to that abcd is not a valid logging level.

3.2 Utility functions

When you think a function might be useful in multiple locations in the project code, it might be a good idea to put it in base/inc/UtilityFunctions.h base/src/UtilityFunctions.cc.

If such function is RISC-V architecture specific, like function implementing an RISC-V specific immediate expansion operation, then put it in riscv/inc/UtilityFunctionsRISCV.h riscv/src/UtilityFunctionsRISCV.cc.

The reason to do this is that although we currently only support RISC-V 64-bit architecture, it is entirely possible to support other architectures in the future, for example, we can relatively easily support RISC-V 32-bit in the future. Other architectures could be support with some amount of work.

The UtilityFunctions files mostly include useful C functions.

3.3 ObjectRegistry

ObjectRegistry is a factory class that hold a template version of the light weight objects inheriting from Object class.

Given a class name, the ObjectRegistry::ObjectInstance method will return a cloned object of the class type.

3.3.1 Register object

Objects inherit from Object class can be registered using ObjectRegistry::RegisterObject method.

In the base layer, all the Object-based classes can be registered inside ObjectRegistry::Setup method in the ObjectRegistry.cc file. For example:

RegisterObject(new RegisterField());

RegisterObject(new BitField());

3.3.2 Obtain a casted instance

Given the inheritance hierarchy, it is sometimes convenient to obtain a cloned object that has already been casted to an intermediate class type.

Suppose we have a Res0RegisterField class that inherit from RegisterField class, the ObjectRegistry::ObjectInstance method will only return an object of type Object, if we want to return the object as the type of RegisterField, we can do this:

string sub_class_type = Res0RegisterField;

const ObjectRegistry *obj_registry =ObjectRegistry::Instance();

RegisterField * new_reg_fld = obj_registry->TypeInstance<RegisterField>(sub_class_type);

The above example is not very useful, but if there are multiple sub-classes of RegisterField and we obtain their name in a variable, we can obtain a predictable RegisterField base class type to operate on.

The TypeInstance<T> method is a template function, where the T can be any type that inherits from Object class.

3.4 Fmt object

The Fmt object makes iostreams formatted I/O more like printf/scanf.   For example, the equivalent to :

    printf(%08X\n, x);

is:

std::cout << fmtX0(x, 8) << \n;

Supported formats include:

fmt:  default formatting

fmtd: decimal

fmtd0: decimal with filling 0

fmto:  octal

fmto0: octal with filling 0

fmtx:  hex

fmtx0: hex with fill 0

fmtX: hex upper case

fmtX0: hex upper case with fill 0

In all cases the arguments for fmtx() are operand, width and precision.  If width and precision are not provided, they default to 0 and 6 respectively (following the normal iostream defaults).

3.4 Enum classes

When using enums, usually we need to add enum-to-string and string-to-enum translation functions.  A script has been added in utils/enum_classes/create_enum_files.py to make this less tedious, we auto-generate those functions. We also generate unit-tests for the Enum classes.

When adding a new Enum type, simply add it in the base_enum_classes.py and do make enums, then base/inc/Enums.h and base/src/Enums.cc will be updated with the new enum type.  The RISC-V specific Enums can be defined in riscv_enum_classes.py.

Adding new enum type is very simple, just add it in the base_enum_classes_details list.

The first list item is the enum type name.  The actual name will have an E prefix added to the provided enum type name.

The second list item is the enum base type, this is a C++11 concept, usually using unsigned char here unless the enum values cannot be contained in unsigned char data type.

The third list item is some descriptions text that will show up as comments in the Enums.h file.

The fourth list item is itself a list, each list item in this list is an enum name and its corresponding value.

4  Unit testing

We use a simple open source project named LEST as the support of our unit-testing framework.  The reasons for picking LEST are its simplicity and easy to use, thus making doing unit-testing easy and even attractive.

4.1 LEST project code

The slogan of LEST is:

Lest Errors eScape Testing

The project site of LEST is here:

https://github.com/martinmoene/lest

Examples from LEST code has been added to: force-riscv/unit_tests/examples. You can do a make in that directory to example the behavior of those tests.

4.2 Setting up a new test directory

To start a unit test directory, use the following command:

cd force-riscv/unit_tests

./template/start_test.py t ModuleName

A directory named ModuleName will be created in force-riscv/unit_tests/tests.  The command will fail if such directory already exists.

When initially created, the directory will contain the following files:

Makefile

Makefile.target

ModuleName_test.cc

The Makefile shouldnt need to be changed.

If more source files (*.cc files) need to be pulled in, list them inside the Makefile.target file assigned to ALL_SRCS variable, in a white-space separated list.

Edit the ModuleName_test.cc source file to add testing details.

Do a make in the tests/ModuleName directory after adding testing details, if passes compiling, an executable will be produced here: tests/ModuleName/bin/ModuleName_test.

Execute tests/ModuleName/bin/ModuleName_test, if every checks pass, the test will pass silently.  Otherwise, error will be reported.

4.3 Example of setting up a test directory

An example has been setup in the force-riscv/unit_tests/InstructionStructure directory.  Please exam the InstructionStructure.cc and Makefile.target files in that directory to inspect details of the changes made to the default template files.

In summary, the changes to Makefile.target are the addition of InstructionStructure.cc, Log.cc, UtilityFunctions.cc, referencing the source code containing functionalities used in the InstructionStructure_test.cc file.

In the InstructionStructure_test.cc file, necessary header files are included, testing/checking are done on a few features of the OperandStructure class.

4.4 Unit test command line options

All unit test executables has a set of command line options provided by the underlying LEST library.  You can run any of the unit test executable with -h option to show all available command line options.

Notably -p option is used to print out test passing statements even when all tests pass.

4.5 Test FAIL calls

In the FORCE-RISCV project code, FAIL calls are used quite a lot to report fatal errors.  In order to test this execution path in unit testing, we have enhanced LEST code and added an EXPECT_FAIL macro.

In a testing scenario, you can setup object TestObjest in a state when calling its method TestMethod will result in a FAIL(test-object-fail) call, which in non-unit-testing environment will exit the program with error message containing test-object-fail emitted.  In unit-testing environment, you can verify that the fail path will be taken by using EXPECT_FAIL macro in the following syntax format:

EXPECT_FAIL(TestObject.TestMethod(), test-object-fail);

5  Module development requirements

The following are the general requirements of developing modules of the FORCE-RISCV project:

         Follow the coding standard as much as possible.

         Comment the code being developed in Doxygen detectable format, while working on the code, not leaving the comments to be done at a later time.

         To check-in a developed module code, the code need to compile cleanly without warning; resulting executable should be able to execute the smoke test without any issue; and there should be written unit tests and they should pass.  One can always check in the initial version with accompanying simple tests, and check in more changes accompanied by more unit testing/checking.

         When found or thought of a potential issue, try to prove it with adding testing/checking in the associated unit-testing code, then fix the module code to pass unit-tests.

         A regression system will be developed over time so that new addition need to pass regression before checking-in.  In phase one proving unit-testing is enough for checking-in until we amass a meaningful regression.

         Try to promote code that you think could be shared/reused to the UtilityFunctions module, or we can consider adding modules for it if the code size is significant.
